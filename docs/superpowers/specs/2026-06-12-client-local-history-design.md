# 客户端本地历史存储设计

**日期：** 2026-06-12  
**范围：** kaya-beam Windows 客户端（Tauri + Vue）本地持久化聊天记录与文件传输历史  
**状态：** 已确认设计，待审阅

---

## 1. 目标

为客户端增加基础版本地历史能力，覆盖两类数据：

1. **聊天记录** — 按“卡雅会话”组织，并允许一个卡雅会话关联多个底层 ACP 会话。
2. **文件传输历史** — 持久化记录发送/接收文件事件，供 UI 展示和后续检索。

本次设计目标是：

- 重启客户端后自动恢复最近的卡雅会话与其最近的 ACP 会话。
- 聊天页左侧边栏展示会话列表。
- 当前会话消息可从本地数据库加载。
- 文件传输历史不再只存在于内存态。
- 在底层重连、ACP session 变更时，用户视角中的“同一段聊天”保持连续。

---

## 2. 非目标

本次不做以下能力：

- 会话搜索
- 自动摘要标题
- 会话重命名
- 删除/归档会话
- 聊天全文检索
- 文件历史高级筛选
- 跨设备同步
- 导入/导出历史
- ORM 或通用存储框架

本次只做**基础可用版**，重点是把数据稳定落地、可恢复、可展示。

---

## 3. 概念模型

### 3.1 卡雅会话（Kaya Session）

卡雅会话是**用户层会话**，是聊天记录的主归属单位。

用户在 UI 中看到、切换、恢复的会话，都是卡雅会话。

特点：

- 一个卡雅会话代表用户眼中的“一段聊天”
- 一个卡雅会话可以关联多个 ACP 会话
- 重连或桥接重建时，不自动切断卡雅会话

### 3.2 ACP 会话（ACP Session）

ACP 会话是**技术层会话**，用于记录后端桥接产生的真实 session 标识。

特点：

- 一个 ACP 会话属于一个卡雅会话
- 同一个卡雅会话下可以先后产生多个 ACP 会话
- ACP 会话用于恢复、调试、追踪，不直接作为 UI 主会话

### 3.3 聊天消息（Chat Message）

每条聊天消息：

- **主归属**于一个卡雅会话
- **可关联**一个 ACP 会话，表示该消息在技术上来源于哪次 session

### 3.4 文件传输历史（File Transfer History）

文件传输历史是独立历史流，不强行嵌套到会话树中。

特点：

- 记录发送/接收文件事件
- 可选关联卡雅会话、ACP 会话
- 即使某次文件传输不属于聊天上下文，也能独立记录

---

## 4. 存储底座选择

本次选择 **SQLite** 作为本地持久化底座。

### 选择理由

- 结构化数据天然包含多实体关系（会话、消息、文件历史）
- 需要按会话查询消息、按时间查询历史
- 后续如果增加分页、搜索、筛选，SQLite 更稳
- 比 JSON 文件更适合 append + query 场景

### 明确不采用的方案

- 通用 KV 存储
- 单 JSON 文件汇总全部历史
- 通用 Repository/Storage 抽象层
- ORM 框架

本项目将采用：

> **一个统一 SQLite 文件作为底层存储，多个目的型模块分别实现聊天历史和文件历史逻辑。**

## 5. 数据表设计

### 5.1 `kaya_sessions`

表示用户层会话。

建议字段：

- `id` TEXT PRIMARY KEY
- `title` TEXT NOT NULL
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL
- `is_active` INTEGER NOT NULL DEFAULT 0

说明：

- `title` 基础版使用简单默认标题，例如“新会话 2026-06-12 18:30”
- `is_active` 表示当前 UI 正在查看的卡雅会话

#### 会话标题基础规则

基础版采用两阶段标题策略：

1. **创建会话时**，先使用默认标题：

   ```text
   新会话 YYYY-MM-DD HH:mm
   ```

2. **当该会话的第一条用户消息发送后**，如果当前标题仍是默认标题，则自动更新为这条用户消息的简短文本。

标题更新规则：

- 去除首尾空白
- 把换行折叠为空格
- 保留原始文本主语义
- 可在 UI 层做省略显示，但数据库中应尽量保留完整或较长版本

基础版明确**不做**：

- 手动重命名
- AI 自动摘要标题

### 5.2 `acp_sessions`

表示某个卡雅会话下的技术会话。

建议字段：

- `id` TEXT PRIMARY KEY
- `kaya_session_id` TEXT NOT NULL
- `remote_session_id` TEXT NOT NULL
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL
- `is_active` INTEGER NOT NULL DEFAULT 0

约束：

- `kaya_session_id` FOREIGN KEY → `kaya_sessions(id)`

说明：

- `remote_session_id` 是真实 ACP session 标识
- 一个卡雅会话可以对应多条 ACP 会话记录

### 5.3 `chat_messages`

表示聊天消息。

建议字段：

- `id` TEXT PRIMARY KEY
- `kaya_session_id` TEXT NOT NULL
- `acp_session_id` TEXT
- `role` TEXT NOT NULL
- `content` TEXT NOT NULL
- `created_at` TEXT NOT NULL

约束：

- `kaya_session_id` FOREIGN KEY → `kaya_sessions(id)`
- `acp_session_id` FOREIGN KEY → `acp_sessions(id)`

说明：

- `kaya_session_id` 必填
- `acp_session_id` 在基础版中建议写入，便于调试和恢复
- `role` 初始只要求支持 `user`、`assistant`

### 5.4 `file_transfer_history`

表示文件收发历史。

建议字段：

- `id` TEXT PRIMARY KEY
- `direction` TEXT NOT NULL
- `file_name` TEXT NOT NULL
- `file_path` TEXT NOT NULL
- `file_size` INTEGER NOT NULL
- `status` TEXT NOT NULL
- `created_at` TEXT NOT NULL
- `kaya_session_id` TEXT
- `acp_session_id` TEXT

说明：

- `direction` 取值：`sent` / `received`
- `status` 基础版取值建议：`ok` / `error`
- `kaya_session_id`、`acp_session_id` 可空

## 6. Rust 端模块边界

### 6.1 新增目的型存储模块

建议新增两个模块，而不是通用存储抽象：

- `chat_history.rs`
- `file_history.rs`

### 6.2 各模块职责

#### `chat_history.rs`

负责：

- 初始化 `kaya_sessions` / `acp_sessions` / `chat_messages`
- 创建卡雅会话
- 绑定/创建 ACP 会话
- 追加聊天消息
- 查询最近会话列表
- 查询某个卡雅会话消息列表
- 恢复最近卡雅会话及其最近 ACP 会话

#### `file_history.rs`

负责：

- 初始化 `file_transfer_history`
- 写入发送/接收文件记录
- 查询最近文件历史

### 6.3 不引入通用存储接口

本次不设计类似以下结构：

- `StorageAdapter`
- `Repository<T>`
- `GenericStore`

原因：

- 目前只有两个具体需求对象
- 它们的查询模型明显不同
- 过早抽象会增加理解和维护成本

## 7. Tauri IPC 设计

前端不直接访问 SQLite，通过 Tauri command 与 Rust 存储层交互。

基础版需要提供的能力：

### 会话相关

- 加载最近卡雅会话
- 加载卡雅会话列表
- 新建卡雅会话
- 切换当前卡雅会话
- 加载某个卡雅会话消息

### 消息相关

- 写入用户消息
- 写入助手消息
- 在运行时为当前卡雅会话绑定新的 ACP 会话

### 文件历史相关

- 加载文件历史
- 记录发送文件事件
- 记录接收文件事件

基础版无需暴露复杂查询参数，先满足当前 UI 使用场景。

## 8. 前端状态与 UI 更新

### 8.1 聊天页布局

聊天页改为：

- **左侧边栏：** 卡雅会话列表
- **右侧主区域：** 当前会话消息 + 输入框

### 8.2 前端 Store 变化

#### `chat` store

从单一内存消息流，扩展为：

- 当前 `kayaSessionId`
- 当前 `acpSessionId`
- 会话列表
- 当前会话消息列表
- 新建会话
- 切换会话
- 启动恢复最近会话

#### `file` store

从纯内存 `history` 扩展为：

- 初始化时从 SQLite 加载历史
- 新文件传输事件到来时写入数据库并更新内存列表

### 8.3 首页 / 文件页

- 首页继续展示最近文件历史
- 文件传输页显示持久化后的完整历史记录

## 9. 基础版行为规则

### 9.1 启动时

客户端启动后：

1. 打开 SQLite 数据库
2. 初始化表结构（若不存在则创建）
3. 查询最近活跃卡雅会话
4. 查询该卡雅会话下最近 ACP 会话
5. 前端默认打开该卡雅会话并加载消息

### 9.2 新建卡雅会话规则

采用已确认规则：

1. 用户手动点“新建聊天”时，创建新的卡雅会话
2. 如果当前没有活跃卡雅会话，则自动创建一个
3. 新建时先写入默认标题 `新会话 YYYY-MM-DD HH:mm`

### 9.3 发送消息

发送用户消息时：

- 若无当前卡雅会话，先创建卡雅会话
- 若无当前 ACP 会话，则在连接建立后绑定一个新 ACP 会话
- 消息写入 `chat_messages`
- 如果这是该卡雅会话的第一条用户消息，且当前标题仍为默认标题，则自动把会话标题更新为该消息的简短文本

### 9.4 接收助手消息

收到助手消息时：

- 追加到当前卡雅会话
- 记录当前对应的 ACP 会话

### 9.5 聊天并发规则（基础版硬约束）

基础版明确采用**单活跃流式会话**模型：

1. 同一时刻只允许一个卡雅会话处于“正在回复”状态。
2. 用户可以切换到其他卡雅会话查看历史，但不能在另一会话中并发发起新的流式回复。
3. 如果当前会话 A 仍在流式回复，用户想在会话 B 中继续提问，必须先满足以下其一：
   - A 的本轮回复自然结束；
   - 用户显式中断 A 的当前回复。
4. 每次发送消息时，系统必须把该轮请求绑定到当时的 `kaya_session_id` 与 `acp_session_id`。
5. 用户切换 UI 当前会话时，**不能改变已经发起请求的归属**。即便用户切走，A 会话中已发起的回复仍必须写回 A，而不是写到当前正在查看的 B。

这条规则的目标是：

- 避免两个卡雅会话时间接近时发生串台；
- 避免流式回复在 UI 切换后误写入错误会话；
- 在基础版中主动限制并发复杂度，优先保证历史归属正确。

### 9.6 ACP 重连/重建

当底层技术 session 变化时：

- **不新建卡雅会话**
- 在当前卡雅会话下新增一条 ACP 会话记录
- 后续消息继续归到当前卡雅会话

### 9.7 文件历史记录

每次发送/接收文件后：

- 写入 `file_transfer_history`
- 若当前存在聊天上下文，则可选写入 `kaya_session_id` / `acp_session_id`

## 10. 错误处理策略

### 10.1 数据库打开失败

- UI 提示“本地历史不可用”
- 聊天与文件传输主流程继续可用
- 前端退回内存态

### 10.2 会话恢复失败

- 自动创建一个新的空卡雅会话
- 不阻塞 UI 初始化

### 10.3 历史写入失败

- 记录日志
- 向 UI 发轻量提示
- 不中断消息发送或文件收发主流程

### 10.4 会话/消息关联异常

- 若 ACP 会话未绑定成功，消息仍可写入卡雅会话
- `acp_session_id` 允许暂时为空

### 10.5 会话切换与流式回复冲突

如果用户在会话 A 回复未结束时切到会话 B：

- UI 可以展示 B 的历史内容；
- 但系统必须继续把 A 正在返回的流式消息写回 A；
- 若用户尝试在 B 发起新消息，基础版必须阻止并给出明确提示，例如“当前已有会话正在回复，请等待结束或先中断当前回复”。

## 11. 迁移与兼容性

当前客户端没有本地 SQLite 历史系统，因此本次是新增能力，不涉及复杂历史迁移。

兼容策略：

- 老用户首次启动时自动建库建表
- 现有 `config.json` 保持原样
- 现有 `~/kaya-transfer` 文件落盘逻辑保持不变
- 文件历史只是新增记录层，不更改现有文件保存路径

---

## 12. 风险与约束

### 风险 1：ACP session 恢复语义不完全稳定

缓解：

- 基础版先把 ACP session 作为技术映射保存
- 不承诺第一版实现复杂“无缝上下文恢复”

### 风险 2：前端状态切换复杂度上升

缓解：

- 保持“当前卡雅会话”作为前端唯一主视角
- 不让 UI 直接操作 ACP 会话

### 风险 3：文件历史与聊天上下文耦合过深

缓解：

- 文件历史保持独立表
- 仅使用可选外键做弱关联

---

## 13. 推荐实现路线

### 路线 A：SQLite + 直接 SQL + 目的型模块

特点：

- 直接使用 SQLite
- 手写建表与 CRUD
- 使用 `chat_history.rs` / `file_history.rs` 做模块隔离

优点：

- 与现有项目风格最一致
- 依赖少
- 容易控制

缺点：

- 要手写查询和映射逻辑

### 路线 B：SQLite + 轻量封装

特点：

- 同样用 SQLite
- 在模块内部增加少量辅助封装

优点：

- 比路线 A 更整洁

缺点：

- 比路线 A 多一层组织成本

### 最终推荐

采用 **路线 A 与路线 B 的折中**：

- 使用 SQLite
- 按聊天历史 / 文件历史拆模块
- 不做 ORM，不做通用存储层

---

## 14. 最终结论

本设计选择：

- **SQLite** 作为统一存储底座
- **卡雅会话 / ACP 会话分层** 作为聊天模型
- **聊天记录主归属卡雅会话**
- **文件传输历史独立存储，可选关联会话**
- **聊天页左侧边栏展示会话列表**
- **启动自动恢复最近卡雅会话与最近 ACP 会话**

这是一版最小但结构正确的基础方案：

- 不过度抽象
- 不锁死未来扩展
- 保持用户体验连续
- 与当前项目代码风格一致
